适用时机：在任何代码改动之前引入；提交或最终回复前再次对照检查。

# 在开始本次代码生成/修改前，先执行“历史错误防复发流程”

## 流程要求：

1. 回看历史反馈
   - 先回顾本对话中我上次指出过的错误、返工点、遗漏点或不满意的地方。
   - 如果历史反馈不够明确，请先用 1-3 条列出你能识别出的风险点，不要凭空扩展。

2. 转成检查清单
   - 把这些历史错误转成可执行的检查项。
   - 每个检查项必须能通过阅读代码、运行测试、查看 diff 或复现步骤来验证。
   - 不要写“注意质量”“避免粗心”这类不可检查的空话。

3. 开始前确认范围
   - 说明本次任务的目标。
   - 说明你计划修改哪些文件或模块。
   - 说明哪些文件或行为不应该被改动。
   - 如果存在高风险假设，先提出来。

4. 执行修改
   - 修改代码时必须遵守上面的检查清单。
   - 如果发现检查清单中的风险实际存在，先修复根因，不要只修表面现象。
   - 不要扩大任务范围，除非明确说明原因。

5. 提交/生成前对照检查
   - 在最终回复或提交前，逐条对照检查清单。
   - 每条都要给出结果：通过 / 失败 / 无法验证。
   - 对“通过”项，给出简短证据，例如文件路径、测试命令、关键逻辑或 diff 说明。
   - 对“失败”或“无法验证”项，先说明原因；能修的先修，不能修的明确残余风险。

6. 最终输出格式
   - 先给出“历史错误检查清单”。
   - 再给出“本次修改摘要”。
   - 再给出“提交前对照检查结果”。
   - 最后给出“测试/验证结果”。
