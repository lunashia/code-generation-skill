适用时机：在开始生成或修改代码时引入。

## 代码完整可运行：

- 明确入口文件和运行方式
- 如需环境变量，请一并说明默认值和示例
- 如果有多个文件，请按文件名分块输出
- 代码必须可启动、可运行、可验证
- 不要只给片段，必须补齐所有缺失文件、依赖、配置和入口
- 默认环境、版本、安装命令、启动命令要写清楚
- 不要假设用户已经有某个依赖或全局配置
- 生成前先自检一遍，尽量避免语法错误、路径错误、导入错误、未定义变量
- 如果有无法确认的地方，先采用合理默认，不要留 TODO
- 最后请列出按什么步骤运行
- 如果不能保证一次就能跑通，请先告诉用户阻塞点，不要先输出看似完整但跑不起来的代码。

## 禁止写TODO：

- 请直接生成可交付代码，不要在代码里留下 TODO、TBD、placeholder、stub、mock 空实现等。
- 如果某处信息不明确，停下来，在生成前向用户提问，不要留待后续处理。
- 所有功能都要给出实际可运行代码。
- 生成前请自查一遍，确保没有未完成标记和半成品逻辑。

## 不引入新依赖：

禁止新增任何第三方依赖、包或框架。
只能使用：
1. 标准库
2. 当前项目已存在的依赖
3. 现有构建工具

### 重要：

- 如果现有依赖无法完成，请先停下来询问用户，不要擅自加包。

## 核对import/include:

请先确认将要使用的库和模块真实存在，再写 import。
不要凭空编造库名、路径名或导入名。
只允许导入：
1. 项目中已存在的依赖
2. 标准库
3. 能明确说明来源和版本的库

如果不确定某个库是否存在，请先停下来说明，不要直接写 import。

- 生成代码前，请先检查所有 import 是否合理、可解析、与当前语言/框架/项目依赖匹配。
- 禁止使用虚构库、猜测性库名、错误路径导入和不存在的命名导出。
- 如果需要新库，请先说明为什么必须用它，并等待确认。

## 确认API真实存在：

请只使用真实存在、可验证的 API。
不要编造函数名、类名、参数名、方法链或返回值。
所有 API 必须来自：
1. 语言标准库
2. 当前项目已安装的依赖
3. 官方文档或已知版本的公开接口

如果不确定某个 API 是否存在，请先说明不确定，不要直接写进代码。

## 禁止any：

- 请避免使用 any。
- 如果类型暂时不明确，请优先用更准确的联合类型、接口、泛型、unknown 或类型守卫。
- 只有在用户明确允许的情况下，才可以使用 any。

## 不要做未经要求的改动和优化：

- 禁止过度设计和未经要求的优化。
- 不要为了“更优雅”擅自改架构、改接口、改数据流、改目录结构。
- 如果认为某个优化值得做，先向用户说明收益、代价和影响，再询问用户是否要做。
