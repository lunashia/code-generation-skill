适用时机：在交付最终答案前引入，用于验收本次代码产物。

不是只负责生成代码，还要负责验收代码。
最终要让用户看到一个可感知结果：这次需求是否完成、改了哪里、怎么验证、还有什么限制。

请按以下流程执行：

1. 提炼用户的真实需求，写成可验收的 checklist。
2. 根据 checklist 实现代码。
3. 自己审查代码是否逐项满足 checklist。
4. 检查是否存在编译错误、类型错误、缺失 import、未定义变量、错误 API 调用。
5. 给出可运行的构建/测试命令。
6. 如果不能实际运行代码，请明确说明，并做静态检查。
7. 如果发现不符合需求或无法编译，请先修复，再给最终答案。

最终答案不要只说“完成了”，必须包含：
- 需求 checklist
- 每项是否完成
- 编译/测试验证方式
- 修改了哪些文件
- 已知限制

## 最终交付单：

最终回答请优先使用下面的结构，除非用户明确要求其他格式：

1. 本次结果
   - 用一句话说明这次交付达成了什么可感知结果。

2. 需求 checklist
   - 用可验收的条目列出用户真正要的结果。
   - 每一项都标明完成 / 未完成 / 无法验证。

3. 修改摘要
   - 只列出和本次需求直接相关的文件和行为变化。
   - 不要描述无关重构、无关优化或未发生的改动。

4. 验证结果
   - 写清楚实际运行过的命令和结果。
   - 如果没有运行，说明原因，并说明做过哪些静态检查。

5. 已知限制
   - 只写真实存在的限制或风险。
   - 如果没有已知限制，就明确写“暂无”。

## 重要：

- 最终交付单要让用户不读代码也能判断本次结果是否符合要求。
- 不要用空泛表述代替验证结果，例如“应该可以”“大概没问题”。
- 如果本次没有修改代码，也要说明分析结论、未修改原因和可继续执行的下一步。
